# v0.8 — Auto-Retry Policy

> **Date:** 2026-03-27
> **Status:** Draft
> **Prerequisite:** v0.7.0 (adaptive execution runtime — feedback loop)

---

## Goal

Reduce manual recovery for retryable failures without weakening human-in-the-loop control.

Auto-retry is an **internal execution loop recovery mechanism** — not an external orchestration step. It lives inside Step C, integrated into the wave dispatch loop, where it checks failed nodes and resets eligible ones before the next iteration.

---

## Motivation

v0.7 introduced the feedback loop: the system learns from past execution outcomes. v0.8 extends this with the second adaptive capability: **recovering from failures automatically**.

The core principle: **the system retries what you would have manually retried** — without removing your ability to control when retries happen.

v0.8 does not make the system autonomous. It removes friction from a specific, bounded failure pattern that already has a clear human-in-the-loop escape hatch (`--no-auto-retry`).

---

## Architecture

### Retry Lives Inside Step C

Auto-retry is integrated into **Step C (DAG Dispatch Loop)**, not implemented as a post-loop hook or external orchestration step.

```
Wave N dispatch
  → node results collected
  → partial_failed detected
  → for each failed node:
       if retryable(node) and retry_count(node) < 1 and auto_retry_enabled:
           reset node to pending
           increment retry_count(node)
  → next iteration of Step C loop picks up reset nodes
```

This means:
- **Execution semantics remain coherent** — user sees `backend failed → retrying... → Wave N+1: re-running`
- **No awkward intermediate graph status** — retry is processed within the loop, not after it exits
- **Consistent with existing runtime architecture** — `advance_node()`, `get_ready_nodes()`, wave iteration all already exist

## Schema Changes

### Node Schema (v0.8)

Two new fields on each node, written by `build_execution_graph()`:

```json
{
  "node_id": "backend-1",
  "team": "backend",
  "depends_on": ["product-1"],
  "status": "pending",
  "artifacts_in": ["product/task-123/prd.md"],
  "artifacts_out": ["backend/task-123/api-design.md"],
  "retryable": true,
  "retry_count": 0
}
```

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `retryable` | boolean | `true` | Whether this node is eligible for auto-retry. Written explicitly by `build_execution_graph()` for every node. |
| `retry_count` | integer | `0` | Number of retries already attempted. Written explicitly — no field-missing-as-zero. |

**Design decision:** Both fields are stored directly on the node record, not in a graph-level map. This keeps execution state local to the node, matches existing runtime design patterns, and makes replay/debug inspection straightforward.

### Execution Graph Metadata

The global auto-retry flag is stored in graph metadata (not in task-level runtime state), so it is readable in replay/debug:

```json
"execution_graph": {
  "metadata": {
    "graph_template": "linear-pipeline",
    "auto_retry_enabled": true
  }
}
```

## CLI Control

### `--no-auto-retry`

Disables auto-retry for this run. Passed at invocation time only — does not persist to task JSON for future runs.

```bash
@orchestrator --no-auto-retry "Build a SaaS landing page"
```

When present:
- `auto_retry_enabled = false` in `execution_graph.metadata`
- No retry eligibility checks fire regardless of `retryable` or `retry_count`
- Existing runtime behavior (manual `--resume`) is unaffected

## Step C Retry Flow

### Insertion Point

After collecting node outcomes from a wave dispatch and before the next `get_ready_nodes()` call.

### Logic

```
for each node in wave_nodes:
    if node.status == "failed":
        if auto_retry_enabled
           and node.retryable == true
           and node.retry_count < 1:
            node.status = "pending"
            node.retry_count += 1
        # else: leave status as failed
```

### Guard Conditions (ALL must be true)

| Guard | Check |
|-------|-------|
| Auto-retry enabled | `auto_retry_enabled == true` (from graph metadata) |
| Node is retryable | `node.retryable == true` |
| Retry budget available | `node.retry_count < 1` (hard limit: exactly one retry per node) |
| Failure type | Only triggers on `failed` nodes; does not affect `blocked` or `completed` |

### Key Behavioral Rules

1. **Retry happens in the next loop iteration, not immediately** — the node is reset to `pending` and picked up by `get_ready_nodes()` on the following iteration. This avoids same-wave immediate re-dispatch.

2. **Only one retry per node** — `retry_count < 1` is a hard limit. A node that fails after a retry is not retried again.

3. **Does not fire on `blocked`** — blocked nodes wait for dependencies, not retried.

4. **`--no-auto-retry` overrides all retry logic** — if the flag is present, no retry checks fire regardless of node state.

---

## Final Status Semantics

These rules determine `graph.status` after Step C completes:

| Condition | Final Status |
|-----------|--------------|
| All nodes reach `done` (including nodes that failed then recovered via retry) | `completed` |
| At least one node is `failed` and cannot be retried (or retry exhausted) | `partial_failed` |
| At least one node is `blocked` | `blocked` |

**Critical rule:** `partial_failed` describes the **terminal state** — whether unrecoverable failures remain — not a history of past failures. A node that failed, retried, and succeeded results in `completed`, not `partial_failed`.

The `retry_count` field on each node preserves the retry history for replay/debug inspection.

## Orchestrator Integration

### Flag Parsing (Step 0)

In Step 0, after existing flag parsing:

```
**If `--no-auto-retry` appears:**
- `auto_retry_enabled = false`
- Strip `--no-auto-retry` from the input.
- Set `execution_graph.metadata.auto_retry_enabled = false`.
- Default: `auto_retry_enabled = true` (if flag absent).

Otherwise:
- `auto_retry_enabled = true`
```

### build_execution_graph Modification

Add `retryable: true` and `retry_count: 0` to every node in all template branches (`data-first`, `research-branch`, `linear-pipeline`, `flat-parallel`).

### Step C Modification

Insert retry eligibility check after wave dispatch results are collected and `advance_node()` has updated node statuses:

```
# After: node results collected, advance_node() called
# Before: next get_ready_nodes() call

if auto_retry_enabled:
    for node in failed_nodes_this_wave:
        if node.retryable and node.retry_count < 1:
            node.status = "pending"
            node.retry_count += 1
            # Log: "retrying <node_id> (retry_count: 0 → 1)"
```

### Retry Debug Output

When `--debug` is active and a retry fires:

```
[node_id]:
  result: failed
  retryable: true
  retry_count: 0 → 1
  action: reset to pending, retrying in next wave
```

## Feedback / Event Log

v0.8 does **not** write a separate retry event to `execution_history.jsonl`.

Retry is an intra-run recovery mechanism, not a terminal outcome. The final outcome is what matters for statistics.

However, to support future analysis, the `final_status` event (written on terminal state) may be extended in a future version:

```json
{
  "task_id": "task_abc",
  "retried_nodes": ["backend-1"],
  "final_status": "completed"
}
```

This is **out of scope for v0.8** but the schema design should not preclude it.

---

## Out of Scope

- **Multiple retries per node** — v0.8 is exactly one retry per eligible node
- **Retry delay** — no wait/sleep between retry and re-dispatch
- **Retry on failure type** — only `failed` status triggers retry; `blocked` does not
- **Auto-retry in dispatch_policy** — retry control is not a dispatch policy concern (separate from routing/selection policy layer)
- **Per-node retry budget > 1** — `retry_count < 1` hard limit; future versions may offer `max_retries: N` if needed
- **retry_reason or failure_category** — failure type analysis out of scope
