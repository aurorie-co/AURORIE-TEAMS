# v0.7 — Adaptive Execution Runtime

> **Date:** 2026-03-27
> **Status:** Draft
> **Prerequisite:** v0.6.0 (persistent execution runtime — replay + resume)

---

## Overview

v0.7 introduces **adaptive behavior** to AURORIE TEAMS: the system begins using historical execution outcomes to bias future routing and template selection decisions.

The core principle: **feedback is additive, not authoritative**. Historical data influences decisions without overriding routing rules or eliminating human control.

### What v0.7 is NOT

- Not an ML system — no model training, embeddings, or probabilistic routing
- Not an autonomous execution system — no auto-retry (deferred to v0.8)
- Not a hard filter — bias is soft, explainable, and conservative

### What v0.7 IS

- A rule-based feedback system using simple aggregations
- An additive bias layer on top of existing deterministic routing
- A conservative system that requires minimum sample thresholds before applying bias
- Fully explainable: every bias adjustment is traceable in debug output

---

## Motivation

AURORIE TEAMS has progressed through:

- **v0.3–v0.5**: Decision → Execution → Coordination
- **v0.6**: Persistence (replay + resume)

v0.6's execution trace produces rich data about team performance and template outcomes — but this data was not yet used. v0.7 turns historical execution outcomes into actionable bias for future routing decisions.

---

## Adaptive Runtime Principles

v0.7 introduces adaptive behavior under strict constraints:

1. **Feedback is additive, not authoritative**
   — Historical data biases decisions, but never overrides routing rules

2. **Learning is explainable**
   — All adjustments (bias multiplier, score changes) must be traceable in debug output

3. **Learning is conservative**
   — Insufficient data (runs < 5) produces no bias; all bias multipliers default to 1.0

4. **Human-in-the-loop is preserved**
   — No automatic execution changes (retry, override) in v0.7

## Data Architecture

### Two-Layer Design

| Layer | File | Role |
|-------|------|------|
| **Source of truth** | `task JSON` | Current state of each task (unchanged) |
| **Event log** | `.claude/workspace/execution_history.jsonl` | Append-only history of execution outcomes |

This separation ensures:
- Task JSON remains clean and focused on current state
- Event log is append-only and optimized for aggregation
- No schema pollution of task JSON with aggregated feedback fields

### Feedback Event Schema

```json
{
  "task_id": "task_abc123",
  "run_id": "task_abc123_run_1",
  "run_kind": "initial",
  "timestamp": "2026-03-27T10:20:00Z",
  "teams": ["product", "backend", "frontend"],
  "graph_template": "linear",
  "final_status": "completed",
  "failed_nodes": [],
  "resumed": false,
  "milestone_id": "ms_xyz"
}
```

**Field definitions:**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `task_id` | string | Yes | Reference to the task |
| `run_id` | string | Yes | Globally unique per run: `{task_id}_run_{n}` where n starts at 1 |
| `run_kind` | string | Yes | `"initial"` or `"resume"` |
| `timestamp` | ISO8601 | Yes | When the event was recorded |
| `teams` | array[string] | Yes | Teams involved in this task execution |
| `graph_template` | string | Yes | Which template was used (e.g., `"linear"`, `"data-first"`, `"flat"`) |
| `final_status` | string | Yes | One of: `"completed"`, `"partial_failed"`, `"blocked"` |
| `failed_nodes` | array[string] | No | Node IDs that failed (empty if none) |
| `resumed` | boolean | Yes | Whether this task was resumed before reaching this state |
| `milestone_id` | string | No | If attached to a milestone |

**run_id generation rule:**
```
run_id = f"{task_id}_run_{n}"
where n = 1 for initial run, incremented for each subsequent resume
```

**Success definition (v0.7, Phase 1):**
- `final_status = "completed"` → **success**
- `final_status = "partial_failed"` → **failure**
- `final_status = "blocked"` → **failure**

**Statistics inclusion rule:**
> Only `run_kind = "initial"` events count toward success_rate statistics.
> Resume runs are excluded to prevent inflated success rates from recovery paths.

## Write Rules

### Trigger Point

**Write exactly one feedback event whenever a task enters a terminal state during a single run.**

Terminal states: `completed`, `partial_failed`, `blocked`

### Constraints

1. **Only on terminal transition** — do NOT write when task is `pending`, `in_progress`, or `awaiting_dispatch_decision`
2. **Exactly once per run** — use an in-memory guard to prevent duplicate writes within the same orchestrator invocation
3. **Multiple runs allowed** — a task that runs to partial_failed then resumes to completed produces two events:
   - `run_kind: "initial"` (partial_failed)
   - `run_kind: "resume"` (completed)

### Orchestrator Integration

```python
# Pseudocode: terminal state hook
def maybe_append_feedback_event(task):
    if not is_terminal_state(task):
        return
    if event_already_written_for_current_run(task):
        return
    event = build_feedback_event(task, run_kind=determine_run_kind(task))
    append_event(HISTORY_PATH, event)
```

Call `maybe_append_feedback_event()` at:
- Normal execution loop end (task reaches terminal state)
- Resume execution loop end (task reaches terminal state)

## Bias Computation

### Team Bias Multiplier

```python
def feedback_multiplier(success_rate: float, runs: int) -> float:
    """
    Returns a bias multiplier in [0.6, 1.0].
    Runs < 5: insufficient data, returns 1.0 (no bias applied).
    """
    if runs < 5:
        return 1.0
    if success_rate >= 0.8:
        return 1.0
    if success_rate >= 0.6:
        return 0.9
    if success_rate >= 0.4:
        return 0.75
    return 0.6
```

**Example:**
- backend: 12 runs, 7 successes → success_rate = 0.58 → multiplier = 0.9

### Graph Template Bias

Same bucket logic applied to template selection. Only applies when **multiple template candidates are available** — never overrides a deterministic single-template choice.

### Apply Location (Routing Pipeline)

```
routing rules → raw_score
feedback stats → bias_multiplier (from JSONL)
raw_score × bias → adjusted_score
adjusted_score → confidence band → dispatch policy
```

**Note:** Bias operates at the score level, NOT at the confidence-band level. This preserves explainability — the raw_score and adjusted_score are both visible.

## Out of Scope for v0.7

- **Auto-retry policy** — deferred to v0.8 (raises human-in-loop concerns)
- **Hard filters** — no team/template elimination based on feedback
- **Weighted sampling** — no probabilistic selection
- **Node-level event tracking** — task-level is sufficient for Phase 1
- **Per-team JSON files** — single JSONL is the right abstraction for v0.7
- **Real-time aggregation caching** — always compute from JSONL at query time (pure functions)

## Orchestrator Integration Points

### New CLI Flag

```bash
--feedback              # Show feedback bias in debug output
--feedback-history      # Print recent feedback events from JSONL
```

### Modified Execution Flow

```
Step 0: parse mode (+ --feedback flag)
Step A: routing (score candidates)
Step B: dispatch policy (apply policy)
Step C: dispatch loop
  → advance_node per result
  → check terminal state → maybe_append_feedback_event()
Step D: milestone update (existing)
```

### apply_team_bias Integration

In Step A (routing), after scoring candidates and before dispatch:

```python
# Load historical feedback
events = load_events(HISTORY_PATH)
stats = aggregate_team_stats(events)

# Apply bias to candidates
candidates = apply_team_bias(candidates, stats)

# Continue with dispatch policy...
```
