# v0.4 — Interactive Routing Contract + DAG Execution

**Status:** Design draft — not yet implemented

---

## Summary

v0.4 expands the decision engine in two directions:

1. **Interactive routing contract** — formalizes `ask` as a first-class system state, not a CLI side effect
2. **DAG execution** — replaces flat team dispatch with dependency-ordered graph execution

The execution order is deliberate: interactive routing must be closed before DAG can work, because DAG's inputs are the finalized `selected_teams`.

---

## Phase 1 — Full Interactive Routing Contract

### Problem Statement

The current `ask` mode is an MVP:

- Only medium band is prompted
- Only yes/no is supported
- CLI-blocking `input()` call
- One-shot — no pause, no resume
- `ask_required: true` is a boolean flag, not a structured payload

This is enough to ship v0.3, but not enough to support UI or API clients, replay, or selective team confirmation.

### Core Design Principles

1. **`pending_decision` is a system state, not a CLI prompt** — CLI renders it as a prompt; UI renders it as buttons; API returns it as a payload. The contract is unified.
2. **Tasks with pending decisions are parked, not failed** — execution halts at Step 5.5 and resumes when the decision is resolved.
3. **Resolution is idempotent and order-independent** — the same decision resolved twice produces the same outcome.

### Step 1 — Upgrade `ask_required` to `pending_decision`

Replace the boolean `ask_required: true` in `routing_decision` with a structured object:

```json
"pending_decision": {
  "type": "dispatch_confirmation",
  "band": "medium",
  "context": "when_no_high_exists",
  "teams": [
    { "team": "product", "score": 2, "confidence": "medium" },
    { "team": "frontend", "score": 1, "confidence": "medium" }
  ],
  "options": ["all", "none"],
  "default": "none"
}
```

**Schema fields:**

| Field | Type | Description |
|-------|------|-------------|
| `type` | string | Always `"dispatch_confirmation"` for now; extensible for future decision types |
| `band` | string | `"medium"` — which confidence band triggered this |
| `context` | string | `"when_high_exists"` or `"when_no_high_exists"` — the medium policy context |
| `teams` | array | All teams in the band, each with `team`, `score`, `confidence` |
| `options` | array | Available actions: `["all", "none"]` in v0.4-a; `["all", "none", "selective"]` in v0.4-b |
| `default` | string | `"none"` — what happens if the user ignores the prompt |

**Changes to existing behavior:**

- `ask_required` is removed in favor of `pending_decision`
- `ask_resolution` is also removed — resolution is now expressed through the task resume interface (Step 3 below)

**Backward compatibility:** v0.3 tasks that have `ask_required: true` can be treated as having `pending_decision` with `options: ["all", "none"]` and `default: "none"` — a one-line migration.

---

### Step 2 — Add `awaiting_dispatch_decision` Task Status

Extend the task status enum. Current statuses:

- `pending` — task created, not yet dispatched
- `needs_clarification` — fallback: no candidates, no ask triggered
- `user_declined_dispatch` — fallback: user said no to ask
- `in_progress` — team lead is executing
- `completed` — all artifacts written

New status:

- **`awaiting_dispatch_decision`** — routing completed, policy triggered ask, task is parked waiting for user decision

**Distinct from `needs_clarification`:**

```
needs_clarification              = system does not know how to interpret the task
awaiting_dispatch_decision       = system knows what to do, user must confirm
```

**Transition:**

```
Step 5.5 → pending_decision is set → task status = "awaiting_dispatch_decision" → stop
```

---

### Step 3 — Resume / Resolve Interface

A unified interface for resolving pending decisions, agnostic to input channel:

```json
// Confirm all
{
  "task_id": "<task-id>",
  "action": "resolve",
  "decision": {
    "type": "dispatch_confirmation",
    "selected": "all"
  }
}

// Decline all
{
  "task_id": "<task-id>",
  "action": "resolve",
  "decision": {
    "type": "dispatch_confirmation",
    "selected": "none"
  }
}

// Selective (v0.4-b only)
{
  "task_id": "<task-id>",
  "action": "resolve",
  "decision": {
    "type": "dispatch_confirmation",
    "selected": "selective",
    "teams": ["product"]
  }
}
```

**Orchestrator behavior on resolve:**

1. Read task JSON, find `pending_decision`
2. Apply `decision` to recompute `selected_teams` and `ignored_teams`
3. Clear `pending_decision`
4. Set task status back to `pending` (or directly into `in_progress`)
5. Continue from Step A/B

**Idempotency:** Resolving a task that is not in `awaiting_dispatch_decision` status is a no-op. Resolving twice with the same payload is also a no-op (idempotent).

**ACL note:** Any caller with write access to `.claude/workspace/tasks/<task-id>.json` can resolve. In v0.4 this is the local Claude Code instance only; auth is out of scope.

### v0.4-a Implementation: All-or-None Confirm

`pending_decision.options = ["all", "none"]`

CLI renders:

```
Medium-confidence teams identified:
- product (score 2)
- frontend (score 1)
Dispatch these teams? [Y/n]
```

- `y` / `yes` / `<empty>` → `selected: "all"`
- `n` / `no` → `selected: "none"`

Same behavior as v0.3 ask mode, but now backed by structured state.

---

### v0.4-b Implementation: Selective Inclusion

`pending_decision.options = ["all", "none", "selective"]`

CLI renders:

```
Medium-confidence teams identified:
- product (score 2)
- frontend (score 1)
Options: [A]ll / [N]one / [S]elective
Select teams (comma-separated, or press Enter for none): product, frontend
```

**Resolution:**

- `selected: "all"` — all medium teams added to `selected_teams`
- `selected: "none"` — all medium teams added to `ignored_teams`
- `selected: "selective", teams: ["product"]` — only product in `selected_teams`, frontend in `ignored_teams`

UI renders as checkboxes. API returns the same three options.

### Phase 1 State Machine (Complete)

```
route → classify → apply policy
  → if ask needed:
      write pending_decision
      status = "awaiting_dispatch_decision"
      park task
      STOP
  → else:
      continue to Steps A/B

user resolves → read task
  → apply decision to recompute selected/ignored
  → clear pending_decision
  → status = "in_progress"
  → execute Steps A/B
```

## Phase 2 — Task Graph / DAG Execution

### Why Phase 2 Must Follow Phase 1

Interactive routing determines **which nodes** are in the graph.
DAG execution determines **the order and dependencies** between them.

If we built DAG first, we'd still face: "the graph is ready but the user wants to remove a node" — which is exactly what Phase 1 solves.

### Design Principles

1. **Static templates first, dynamic synthesis later** — v0.4 templates are hardcoded patterns. v0.5+ can synthesize graphs from artifact contracts.
2. **No automatic retry or cascading failure in v0.4** — upstream failure blocks downstream. Graph status = `partial_failed`. Recovery is manual.
3. **Artifacts are the handoff medium** — `artifacts_in` and `artifacts_out` are explicit paths, not inferred.

### Graph Node Schema

```json
{
  "node_id": "backend-1",
  "team": "backend",
  "depends_on": ["product-1"],
  "status": "pending",
  "artifacts_in": ["product/<task-id>/prd.md"],
  "artifacts_out": ["backend/<task-id>/api-design.md"]
}
```

| Field | Type | Description |
|-------|------|-------------|
| `node_id` | string | Unique within graph, format `{team}-{seq}` |
| `team` | string | Team identifier, maps to workflow file |
| `depends_on` | array of node_ids | Nodes that must complete before this one starts |
| `status` | enum | `pending` / `ready` / `running` / `done` / `failed` / `blocked` |
| `artifacts_in` | array of paths | Input artifact paths (written by depends_on nodes) |
| `artifacts_out` | array of paths | Output artifact paths produced by this node |

### Graph Schema

```json
"execution_graph": {
  "nodes": [
    { "node_id": "product-1", "team": "product", "depends_on": [], "status": "pending", ... },
    { "node_id": "backend-1", "team": "backend", "depends_on": ["product-1"], "status": "pending", ... },
    { "node_id": "frontend-1", "team": "frontend", "depends_on": ["backend-1"], "status": "pending", ... }
  ],
  "edges": [
    ["product-1", "backend-1"],
    ["backend-1", "frontend-1"]
  ]
}
```

### Graph Execution Rules

**Ready condition:** A node is `ready` when all nodes in `depends_on` are `done`.

**Parallel condition:** Nodes with no dependency relationship (disjoint subgraphs) can run in parallel. The orchestrator dispatches all `ready` nodes simultaneously via Step B (parallel dispatch).

**Blocked condition:** If any `depends_on` node is `failed`, this node is `blocked`. Graph status = `partial_failed`.

**Artifact handoff:** Before a node starts, the orchestrator confirms all `artifacts_in` paths exist. If any are missing, the node is `blocked` instead of `ready`.

### v0.4 Static Graph Templates

Three templates only:

**Linear pipeline** (most common):
```
product → backend → frontend
```

**Research branch**:
```
research → product → backend
              ↓
           frontend
```

**Data-first**:
```
data → research → product → backend
```

**Template selection rules:**

| Condition | Template |
|-----------|----------|
| `product` selected, `backend` selected, `frontend` selected | Linear pipeline |
| `research` selected without `product` | Research branch |
| `data` selected, `research` not selected | Data-first |
| No pattern matches | Fall back to v0.3 flat parallel dispatch |

**In v0.4, template selection is automatic and deterministic.** Dynamic synthesis (v0.5+) will use artifact contracts to infer the graph.

### DAG Execution Flow

```
Step 5.5 → selected_teams finalized
  → select graph template
  → build execution_graph with nodes + edges
  → for each node in topological order:
      if node is ready:
          dispatch via Step B (parallel)
          wait for completion
          update node status
      if any node failed:
          mark graph as partial_failed
          stop further dispatches
          output blocked nodes
```

### Graph Status Values

| Status | Meaning |
|--------|---------|
| `pending` | Graph created, no nodes started |
| `in_progress` | At least one node is `running` |
| `completed` | All nodes are `done` |
| `partial_failed` | At least one node `failed`, downstream nodes `blocked` |

### DAG + Interactive Routing Interaction

When `awaiting_dispatch_decision` is resolved (Phase 1):

```
user resolves → selected_teams finalized
  → build execution_graph
  → dispatch first wave of ready nodes
```

When a `pending_decision` exists in a DAG context:

```
graph has product-1 (done), backend-1 (ask triggered)
  → product-1 artifacts are confirmed
  → backend-1 is in await state
  → frontend-1 is blocked (depends_on backend-1 not done)
  → graph status = "in_progress" (not "partial_failed" — it's paused, not failed)
```

After resolution:

```
backend-1 resolved → selected_teams updated
  → backend-1 dispatched
  → frontend-1 still blocked until backend-1 done
```

## v0.4 Complete Architecture

```
┌─────────────────────────────────────────────────────┐
│  Step 0  → parse flags (debug, dry-run, resolve)   │
│  Step 1  → read routing.json, normalize policy      │
│  Step 2  → score teams                              │
│  Step 3  → filter candidates                        │
│  Step 3.5 → sort                                    │
│  Step 4  → assign confidence bands                   │
│  Step 5  → classify (high/medium/none)              │
│  Step 5.5 → apply dispatch_policy                   │
│    if auto → add to selected_teams                  │
│    if ignore → add to ignored_teams                │
│    if ask → write pending_decision                 │
│              set status = awaiting_dispatch_decision │
│              park task                             │
│              STOP                                  │
│  Step 6  → fallback (needs_clarification / declined)│
│  Step 7  → write routing_decision + execution_graph │
│  Step 7.5 → debug trace                            │
│  Step 8  → user summary                            │
│  Steps A/B → dispatch teams (dag-aware)             │
└─────────────────────────────────────────────────────┘

Resolution path:
  receive resolve request
    → read pending_decision from task JSON
    → apply decision to recompute selected/ignored
    → build / update execution_graph
    → continue from Step A/B
```
