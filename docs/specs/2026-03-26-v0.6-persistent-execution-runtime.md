# v0.6 — Persistent Execution Runtime

> **Date:** 2026-03-26
> **Status:** Draft — for alignment before implementation
> **Prerequisite:** v0.5.0 (milestone + selective routing + DAG execution)

---

## Overview

v0.6 adds the time dimension to AURORIE TEAMS: making execution **replayable** and **resumable**.

Today the system can coordinate work across tasks (v0.5) but treats each execution as ephemeral. Once a DAG completes or fails, the execution context is lost. v0.6 changes this by persisting the execution state and making it inspectable and recoverable.

### Core capabilities

1. **Replay** (read-only) — inspect what happened in a past task execution
2. **Resume** — continue DAG execution from where it left off (partial/complete failure)

---

## Motivation

### Why this is the natural next step

AURORIE TEAMS has a complete execution trace in the task JSON:

- `routing_decision` — what was decided and why
- `pending_decision` — whether a human decision was deferred
- `execution_graph` — nodes, edges, wave order, node statuses

These are written to disk but not leveraged for recovery. v0.6 turns them into a persistent runtime capability.

### Competitive context

Most agent systems (Claude subagents, agency-agents, workflow runtimes) treat execution as one-shot. Replay/Resume is a capability that enterprise agent frameworks charge for separately.

### Future unlock: optimization loop

Replay/Resume establishes the foundation for:
- **Failure recovery:** automatic retry of failed nodes
- **Optimization:** comparing execution traces across runs
- **Audit log:** enterprise-grade accountability
- **Behavioral learning:** training data from execution history

## CLI Contract

### Replay

```bash
@orchestrator --replay <task-id>
```

**Behavior:**
- Load task JSON from `.claude/workspace/tasks/<task-id>.json`
- Print routing_decision (selected, secondary, ignored teams)
- Print execution_graph timeline (wave order, node statuses, timestamps)
- Print milestone ref if present
- Exit — no state mutation, no dispatch

**Output format:**

```
=== REPLAY: <task-id> ===

Prompt: <task prompt>
Status: <task status>

Routing:
  Selected:  product, backend, frontend
  Secondary: market
  Ignored:   support, data

Execution Graph:
  Wave 1: [product-1]        → done        2026-03-26 10:01:01
  Wave 2: [backend-1]         → done        2026-03-26 10:02:33
  Wave 3: [frontend-1]        → completed   2026-03-26 10:04:12

Final status: completed
Milestone: Launch SaaS (ms_abc123)

=== END REPLAY ===
```

### Resume

```bash
@orchestrator --resume <task-id>
```

**Behavior:**
- Load task JSON
- Validate `execution_graph` exists and `status` is `in_progress`, `partial_failed`, or `blocked`
- Skip Steps 1–7 entirely (routing already done)
- Enter Step C dispatch loop at current wave
- Dispatch all `pending` or `blocked` (now unblocked) nodes
- Continue until graph reaches terminal state (`completed` | `partial_failed`)

**Resume from partial failure:**
- If `status = partial_failed`: show partial failure summary, prompt for confirmation before retrying
- If confirmed: reset failed nodes to `pending`, re-enter dispatch loop

**Resume from blocked:**
- If `status = blocked`: show blocked nodes + missing artifacts_in, prompt for confirmation
- If confirmed: clear `blocked` flag, re-enter dispatch loop

### Resume — no-op cases

| Condition | Behavior |
|-----------|----------|
| `status = completed` | No-op; print "task already completed" |
| `status = pending` (no graph) | No-op; print "no execution graph to resume" |
| `status = awaiting_dispatch_decision` | No-op; print "task awaiting human decision — use --resolve" |
| `status = user_declined_dispatch` | No-op; print "task was declined by user" |

## Orchestrator Step 0 Extension

```python
# Step 0: parse mode
if "--replay" in args:
    mode = "replay"
    task_id = extract_task_id(args)
elif "--resume" in args:
    mode = "resume"
    task_id = extract_task_id(args)
elif "--resolve" in args:
    mode = "resolve"
    ...
else:
    mode = "normal"
```

### Replay path

```
Step 0: parse --replay
Step R1: load task JSON
Step R2: print routing_decision summary
Step R3: print execution_graph timeline
Step R4: print milestone ref if present
Step R5: exit (no dispatch)
```

### Resume path

```
Step 0: parse --resume
Step C: enter dispatch loop (skip Steps 1-7)
  - load execution_graph
  - get_ready_nodes
  - if empty: check graph status → terminal or blocked
  - dispatch ready nodes
  - advance_node per result
  - repeat until terminal
Step X: update task JSON, print final status
```

## Data Model

### execution_graph — new optional fields

```python
{
  "nodes": [
    {
      "node_id": "backend-1",
      "team": "backend",
      "depends_on": ["product-1"],
      "status": "done",          # pending | running | done | failed | blocked
      "artifacts_in": ["..."],
      "artifacts_out": ["..."],
      "started_at": "2026-03-26T10:02:33Z",   # new
      "completed_at": "2026-03-26T10:02:58Z"   # new
    }
  ],
  "edges": [...],
  "status": "completed",         # pending | in_progress | completed | partial_failed | blocked
  "started_at": "...",           # new
  "completed_at": "...",          # new
  "wave_order": ["product-1", "backend-1", "frontend-1"]  # new
}
```

**Note:** `started_at` / `completed_at` / `wave_order` are optional in v0.6 — existing task JSONs don't have them. Replay must handle missing fields gracefully.

## Milestone Interaction

- Replay does NOT update milestone status
- Resume: after graph reaches terminal state, trigger milestone re-aggregation (same as normal completion)
- Milestone status updates are independent of replay/resume

## Milestone Re-aggregation on Resume Completion

After resume reaches terminal state (`completed` | `partial_failed`):

1. Orchestrator calls `aggregate_milestone_status(tasks_statuses)`
2. Updates milestone JSON: `.claude/workspace/milestones/<id>.json`
3. Milestone status reflects latest state

- **Cross-task resume** — resume a milestone's entire task graph (v0.8+)
- **Execution history storage** — replay-only, single-task JSON (v0.8+ for multi-version history)

## Sequence Diagram

### Replay

```
User: @orchestrator --replay <task-id>
  → Step 0: parse --replay, extract task-id
  → Step R1: read .claude/workspace/tasks/<task-id>.json
  → Step R2: print routing_decision (selected/secondary/ignored)
  → Step R3: print execution_graph timeline (wave order, node statuses)
  → Step R4: print milestone ref if present
  → Step R5: exit
```

### Resume

```
User: @orchestrator --resume <task-id>
  → Step 0: parse --resume, extract task-id
  → Validate graph status is resumable (in_progress | partial_failed | blocked)
  → Step C: enter dispatch loop
    → get_ready_nodes → dispatch pending nodes
    → advance_node per result
    → repeat until graph status terminal
  → Step X: write updated task JSON
  → Trigger milestone re-aggregation if milestone attached
```
