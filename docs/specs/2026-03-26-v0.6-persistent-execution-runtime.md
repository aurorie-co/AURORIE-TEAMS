## Overview

v0.6 adds the time dimension to AURORIE TEAMS: making execution **replayable** and **resumable**.

Today the system can coordinate work across tasks (v0.5) but treats each execution as ephemeral. Once a DAG completes or fails, the execution context is lost. v0.6 changes this by persisting the execution state and making it inspectable and recoverable.

### Core capabilities

1. **Replay** (read-only) â€” inspect what happened in a past task execution
2. **Resume** â€” continue DAG execution from where it left off (partial/complete failure)

---

## Motivation

AURORIE TEAMS has a complete execution trace in the task JSON:

- `routing_decision` â€” what was decided and why
- `pending_decision` â€” whether a human decision was deferred
- `execution_graph` â€” nodes, edges, wave order, node statuses

These are written to disk but not leveraged for recovery. v0.6 turns them into a persistent runtime capability.

### Future unlock

Replay/Resume establishes the foundation for:
- **Failure recovery:** automatic retry of failed nodes
- **Optimization:** comparing execution traces across runs
- **Audit log:** enterprise-grade accountability
- **Behavioral learning:** training data from execution history

---

## State Priority (Critical Invariant)

When evaluating whether a task is resumable, the following priority rules apply:

1. **`pending_decision` exists** â†’ task is NOT resumable. Must use `--resolve` first.
   - `pending_decision` means the task is parked awaiting human confirmation
   - Resuming would bypass the human decision â€” never allowed
2. **`execution_graph` absent** â†’ task has no DAG to resume
3. **`execution_graph.status`** determines resume behavior (see below)
4. **`task.status`** is a derived user-readable summary, not a control signal

## CLI Contract

### Replay

```bash
@orchestrator --replay <task-id>
```

**Behavior:**
- Load task JSON from `.claude/workspace/tasks/<task-id>.json`
- Print routing_decision (selected, secondary, ignored teams)
- Print execution_graph timeline (wave order, node statuses, timestamps)
- Print milestone ref if present
- Exit â€” no state mutation, no dispatch

**Output format:**

```
=== REPLAY: <task-id> ===

Prompt: <task prompt>
Status: <task status>

Routing:
  Selected:  product, backend, frontend
  Secondary: market
  Ignored:   support, data

Execution Graph:
  Wave 1: [product-1]     â†’ done          2026-03-26 10:01:01
  Wave 2: [backend-1]     â†’ done          2026-03-26 10:02:33
  Wave 3: [frontend-1]   â†’ completed      2026-03-26 10:04:12

Final status: completed
Milestone: Launch SaaS (ms_abc123)

=== END REPLAY ===
```

**Graceful handling of missing fields:** Old task JSONs may not have `started_at`, `completed_at`, or `waves`. Missing timestamps display as `â€”`. Missing `waves` field: reconstruct from node order in graph (nodes with same depends_on depth â†’ same wave).

### Resume

```bash
@orchestrator --resume <task-id>
```

**Pre-conditions (checked in order):**
1. `pending_decision` present â†’ **no-op**, print "task awaits human decision â€” use --resolve"
2. `execution_graph` absent â†’ **no-op**, print "no execution graph found"
3. `execution_graph.status = completed` â†’ **no-op**, print "task already completed"
4. `execution_graph.status = pending` â†’ **no-op**, print "no execution graph to resume"
5. `execution_graph.status = user_declined_dispatch` â†’ **no-op**, print "task was declined by user"

**Resume from `in_progress`:**
- Enter Step C dispatch loop immediately
- `get_ready_nodes` returns currently ready (pending, depends satisfied) nodes
- Dispatch, advance_node per result
- Repeat until graph reaches terminal state

**Resume from `partial_failed`:**
1. Show partial failure summary (which nodes failed)
2. Prompt for confirmation before retrying
3. **On confirm:** reset only `failed` nodes to `pending`; leave `done`, `blocked`, `running` nodes untouched
4. Re-evaluate `get_ready_nodes` â€” only failed nodes that now have satisfied depends are ready
5. Enter dispatch loop for ready nodes

**Resume from `blocked`:**
1. Show blocked nodes + missing `artifacts_in` for each
2. Prompt for confirmation before recovering
3. **On confirm:** re-check `artifacts_in` for each blocked node (verify files exist)
4. Only unblock nodes whose `artifacts_in` are now satisfied; re-enter dispatch loop

**After resume reaches terminal state (`completed` | `partial_failed`):**
- Write updated task JSON
- Trigger milestone re-aggregation if milestone attached

```python
    mode = "normal"
```

### Replay path

```
Step 0: parse --replay
Step R1: load task JSON
Step R2: print routing_decision summary
Step R3: print execution_graph timeline (waves, node statuses, timestamps)
Step R4: print milestone ref if present
Step R5: exit (no dispatch, no state mutation)
```

### Resume path

```
Step 0: parse --resume
Step RV: validate pre-conditions (pending_decision, graph presence, status)
Step RC: dispatch loop (resume from current state)
  - get_ready_nodes â†’ dispatch ready nodes
  - advance_node per result
  - repeat until terminal
Step RX: write updated task JSON
Step RM: milestone re-aggregation if milestone attached
```

## Data Model

### execution_graph

```python
{
  "nodes": [
    {
      "node_id": "backend-1",
      "team": "backend",
      "depends_on": ["product-1"],
      "status": "done",          # pending | running | done | failed | blocked
      "artifacts_in": ["artifacts/product/{task_id}/prd.md"],
      "artifacts_out": ["artifacts/backend/{task_id}/backend-implementation.md"],
      "started_at": "2026-03-26T10:02:33Z",   # optional in v0.6
      "completed_at": "2026-03-26T10:02:58Z"   # optional in v0.6
    }
  ],
  "edges": [["product-1", "backend-1"]],
  "status": "completed",    # pending | in_progress | completed | partial_failed | blocked
  "started_at": "...",      # optional
  "completed_at": "...",    # optional
  # waves: array of arrays, each inner array = one wave (nodes dispatched together)
  # e.g. [["product-1"], ["backend-1", "frontend-1"], ["mobile-1"]]
  # optional in v0.6 â€” if absent, reconstruct from node depends_on depth
  "waves": [["product-1"], ["backend-1", "frontend-1"], ["frontend-2"]]
}
```

**`waves` field semantics:** Each element is a list of node-ids dispatched simultaneously. Ordered by execution sequence. If absent, reconstruct by computing node depth from root (nodes with no depends_on = wave 1, etc.).

**Backward compatibility:** Existing task JSONs may have `wave_order: [...]` (flattened list). Treat `wave_order` as equivalent to a reconstructed waves array where each element is a single-item list in that order.

**`started_at` / `completed_at`:** ISO timestamps. Optional in v0.6 â€” Replay displays `â€”` when missing.

## Milestone Interaction

- Replay does NOT update milestone status
- Resume completion: trigger `aggregate_milestone_status` for attached milestone, write updated milestone JSON
- Milestone status is always derived from task states â€” never a control signal
