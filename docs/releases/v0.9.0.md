# v0.9.0 ‚Äî Verification: Node Done Means Verified Done

> `node = done` is no longer what the model says ‚Äî it is what the system confirms.

---

## Problem

Today, a node's `done` status is determined by the model reporting success. This creates two problems:

1. **Retry is unreliable** ‚Äî a node that reports success but hasn't actually fixed anything will be retried based on false confidence
2. **Feedback is noisy** ‚Äî the feedback loop learns from unverified claims, not from ground truth

---

## Verification Principles

| Principle | Meaning |
|-----------|---------|
| Verification is explicit, not inferred | Only nodes with `verification_command` are verified |
| Verification is binary | Exit code 0 = success; anything else = failure |
| Verification strengthens existing runtime semantics | Verification failure is treated as node failure; no new failure layer |
| Verification is conservative | No multi-command sequences, no dynamic shell behavior, no rule DSL |

---

## Design Principles

1. **Verification command is explicit** ‚Äî only nodes that declare a `verification_command` are verified
2. **Success = exit code 0** ‚Äî no DSL, no expressions, no configuration
3. **Verification failure = node failure** ‚Äî reuses existing `auto_retry` / `partial_failed` / `feedback` semantics, no new failure layer
4. **No schema expansion in v0.9** ‚Äî no `success_condition`, no `verified` field on events, no rule engine

---

## Schema Change

### Node field (additive only)

```json
{
  "node_id": "backend-1",
  "team": "backend",
  "depends_on": [],
  "verification_command": "python3 tests/routing/test_routing_cases.py"
}
```

- Field is **optional**. Absence means no verification (v0.8 behavior unchanged).
- Path must be relative to the **repository root** used by the orchestrator (the working directory where `git`, `tests/`, `artifacts/` etc. are resolved). Explicit and unambiguous ‚Äî not "project root" (too vague).
- Only `python3`, `bash`, `pytest`, `sh` are allowed as the command interpreter.

### Allowed command prefixes (v0.9 hardcoded whitelist)

```
python3 <path> [args...]
pytest <path> [args...]
bash <path> [args...]
sh <path> [args...]
```

- No shell operators (`|`, `&&`, `>` etc.) ‚Äî command must be a single invocation
- No `sudo`, no `rm -rf`, no network calls in verification
- Validation is **token-based** (argv split), not substring-based. A command must start with an exact allowed token; partial prefix matches are not sufficient
- Validation happens at task write time (Step A / Step B), not at dispatch time

**OUT OF SCOPE for v0.9:** dynamic allowlist config, env var interpolation, multi-command sequences.

---

## Orchestrator Step C Change

### Current behavior (v0.8)

```
dispatch node
wait for completion
mark done
```

### New behavior (v0.9)

```
dispatch node
wait for completion

if node execution reports failure:
    mark failed immediately
    do NOT run verification_command   ‚Üê execution failure takes priority
    proceed to auto_retry / partial_failed

if node execution reports success:
    if node.verification_command is absent:
        mark done (v0.8 behavior)
    if node.verification_command is present:
        run verification_command
        if exit_code == 0:
            mark done
        if exit_code != 0:
            mark failed  ‚Üê feeds into auto_retry / partial_failed
```

**Two failure sources must stay distinct:**
- **Execution failure** ‚Äî the dispatched agent/subprocess itself failed; verification does not run
- **Verification failure** ‚Äî execution succeeded but the verification command failed; normal failure path

Mixing these causes dirty failure attribution in feedback events and retry logic.

## Integration with Existing Systems

### Auto-retry (`lib/retry.py`)

`maybe_retry_nodes()` continues to work unchanged. A verification failure produces a normal node failure and is eligible for auto-retry **only if** the node is `retryable: true` and `retry_count < 1` (v0.8 contract). A verification failure does not create a separate retry class ‚Äî it is treated identically to any other node failure.

### Partial failure (`execution_graph.status`)

If a verified node fails verification after exhausting retries, `execution_graph.status = "partial_failed"` ‚Äî same as v0.8.

### Feedback events (`lib/feedback.py`)

v0.9 does NOT add new fields to the feedback event schema. The existing `status` field already captures `completed` vs `failed`. No schema change needed.

**OUT OF SCOPE for v0.9:** `verified: true/false` in event, `verification_exit_code` in event.

## Implementation Tasks

1. **Task 1 ‚Äî Schema + allowlist parser**: Add `verification_command` field to node schema in `orchestrator.md`. Implement token-based allowlist validation at task write time (Step A / Step B). Reject invalid commands before dispatch.
2. **Task 2 ‚Äî Step C executor**: After node reports completion and before marking done, call `run_verification(node)` if `verification_command` is present. Check exit code ‚Äî 0 ‚Üí done, non-zero ‚Üí failed. Do NOT run verification if node execution itself failed.
3. **Task 3 ‚Äî Auto-retry integration**: Connect verification failure to existing `maybe_retry_nodes()` path. A verification failure is a normal node failure ‚Äî eligibility for retry follows existing `retryable` / `retry_count` contract.
4. **Task 4 ‚Äî Integration tests**: Five test cases covering: (a) no `verification_command` ‚Üí v0.8 behavior unchanged; (b) verification exit 0 ‚Üí done; (c) verification non-zero ‚Üí failed; (d) verification fail + retryable + retry_count < 1 ‚Üí retry once; (e) execution failure ‚Üí no verification runs, normal failure path

---

## What Is NOT in v0.9

| Feature | Why excluded |
|---------|-------------|
| `success_condition` DSL | Creates a rule engine ‚Äî not a runtime concern |
| `verified` field on events | Schema bloat before we have feedback data to validate |
| Dynamic allowlist config | Over-engineering before we know what commands teams actually need |
| `verification_exit_code` in event | Premature instrumentation |
| Multi-command sequences (`&&`, `\|`) | Security surface area, execution semantics ambiguous |
| Verification timeout config | Uses the same default execution timeout as node dispatch; explicit value TBD by implementation |

---

## Rollout Strategy

Start with a single team (backend) to validate the mechanism before requiring all teams to define verification commands.

Teams that don't declare `verification_command` continue to work as v0.8 ‚Äî no forced migration.
