## v0.6.0 — Persistent Execution Runtime

v0.6 adds the time dimension to AURORIE TEAMS: making execution **replayable** and **resumable**.

Today the system can coordinate work across tasks (v0.5) but treats each execution as ephemeral. Once a DAG completes or fails, the execution context is lost. v0.6 changes this by persisting the execution state and making it inspectable and recoverable.

### Highlights

#### Replay — execution transparency layer

Read-only inspection of any past task execution:

```
@orchestrator --replay <task-id>
```

Output shows:
- Full routing decision (selected, secondary, ignored teams)
- Execution graph timeline (wave order, node statuses, timestamps)
- Milestone reference if attached

Replay is **auditability built in** — every execution decision and progression is now visible, not buried in logs.

#### Resume — execution is no longer ephemeral

Continue DAG execution from where it left off:

```
@orchestrator --resume <task-id>
```

Supports three recovery paths:

- **`in_progress`** — continues from current wave, dispatching only ready nodes
- **`partial_failed`** — resets only failed nodes (done/blocked/running untouched), then retries
- **`blocked`** — re-checks `artifacts_in` before unblocking, only unblocks nodes whose artifacts now exist

Resume is **human-in-the-loop by design** — every recovery path prompts for confirmation before mutating state.

#### State priority as code

The critical invariant — "never resume a task that awaits a human decision" — is now enforced by `validate_resume()`, a pure function with explicit priority rules:

1. `pending_decision` exists → NOT resumable (must `--resolve` first)
2. `execution_graph` absent → nothing to resume
3. `execution_graph.status` determines behavior

This prevents accidental bypass of human decisions and makes the resume contract explicit and testable.

### Why this matters

AURORIE TEAMS is no longer a "run once" orchestrator.

It now has:

- **Execution memory** — past runs are inspectable (Replay)
- **Execution continuity** — interrupted runs can be continued (Resume)
- **Failure resilience** — partial failures can be retried in isolation
- **Auditability** — wave-by-wave progression is recorded and viewable

This is the foundation for: automatic retry policies, failure pattern learning, and routing feedback loops in future versions.

### What's next

- Graph visualization — visual display of execution graph and wave progression

- Cross-task resume — resume all tasks in a milestone at once

---

**Migration note:** v0.6 is backward-compatible with v0.5 task JSONs. Replay gracefully handles old tasks without `waves`, `started_at`, or `completed_at` fields.
