## v0.5.0 — Goal-Oriented Coordination Runtime

v0.5 extends AURORIE TEAMS beyond single-task orchestration.

This release adds a persistent coordination layer across tasks, plus fine-grained human control over team selection.

### Highlights

#### Milestone coordination

- Create and track milestones across multiple tasks
- Aggregate status: `partial_failed > in_progress > completed > pending`
- Milestone state is independent from routing and execution — decisions made inside a task don't affect milestone tracking
- CLI: `--milestone "Launch SaaS" "Build a crypto trading platform"` creates a milestone and attaches the first task
- CLI: `--milestone-status ms_abc123` queries aggregated status across all attached tasks
- Append-only: tasks can be added to a milestone, never removed

#### Selective interactive routing

- Extend decision resolution from `all | none` to `all | none | selective`
- When a task is parked awaiting confirmation, users can approve only a subset of medium-confidence teams
- `@aurorie-orchestrator --resolve <task-id> selective backend,product`
- Deterministic and idempotent — same resolution twice produces the same result
- Empty selective list is equivalent to declining (sets `declined_after_ask = true`)

#### Existing runtime foundation

- Interactive pause / resolve flow (`pending_decision` + `awaiting_dispatch_decision`)
- Dependency-aware DAG execution with wave-based parallel dispatch
- Structured `routing_decision` and `execution_graph` schemas
- Blocked node handling: missing `artifacts_in` marks a node blocked, graph stops gracefully
- Strong test coverage: **83/83 tests green**

### Why this matters

AURORIE TEAMS is no longer just a task executor.

It is now a **goal-oriented coordination runtime**:
- decisions can pause
- execution can resume
- tasks can roll up into milestones
- human control can be partial, not all-or-none
